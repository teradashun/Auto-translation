# プロジェクト名：arxiv-translator

## 概要：arXiv論文をMarkdown/日本語LaTeXに自動変換するパイプライン
**制作時間**：100時間程度

**使用技術**
- 言語：

1. Python：LLMの組み込みが容易であるため
- フロントエンド：

1. Streamlit：Pythonのみでフロントエンドを完結させ、限られた開発リソースをLLMパイプラインの構築に集中させるため
- バックエンド：

1. FastAPI：LLMのAPI呼び出しというI/O待ち時間の長い処理に対して、非同期処理を容易に実装でき、パフォーマンスを最適化できるため。
2. Docker：ローカル環境で確実に動くものを作り、その後AWSに同様の環境でデプロイできるため。
- インフラ

1. AWS：ローカルのDocker環境で構築したコンテナをそのままデプロイ・稼働できるサービスが存在し、インフラの構築・運用コストを抑えつつ、バックエンドとフロントエンドを分離して安定稼働させることができるため。

**工夫した点**
- LLM出力の揺らぎに対する堅牢な前処理・後処理の実装
    
    生成AIがMarkdownからTeXに変換した際に、Markdown特有の記法が残存するため、数式はそのまま残し、文章中の「_」のみを正規表現を用いて削除しました。図や表も同様に削除し、併せて自動コンパイルの成功率を担保しました。
    
- コスト最適化とレートリミットを考慮したモデル選定
    
    開発コストの最適化を図る上で、レートリミットの厳しいモデル（Gemini-1.5-Flashなど）を使用すると、すぐに上限に達してしまう問題が発生しました。そこで、API制限が緩いモデル（Gemma-3 27B）を使用することで、大量の論文テキストを安定的かつ低コストで処理するパイプラインを確立しました。さらに、Few-shotプロンプティングや出力フォーマットの厳密な定義を行うことで、質を担保しました。
    
- ファイルを複数に分割して翻訳の質を担保
    
    一度に多くの文章を翻訳させると、翻訳の質が落ちたり省略されたりします。ファイルの上限を3000文字に設定することでこれらの問題を解決しました。
    
- 余分なAPI消費を避けるため、すでにファイルが存在する場合は翻訳をスキップ
    
- レート制限が発生した際のリトライ処理
    レート制限が発生した際に即座にエラーを発生させるのではなく、何度かリトライを行うことで、翻訳を完成させる確率を上昇させました。

**実装手順**
1. Dockerがインストールされていることを確認してください。
2. `git clone https://github.com/teradashun/Auto-translation.git`
3. `cp .env.example .env` を実行し、ファイル内の GEMINI_API_KEY をご自身のAPIキーに変更してください
4. `docker compose up -d`
5. 起動後にブラウザで「http://localhost:8501」へアクセス
